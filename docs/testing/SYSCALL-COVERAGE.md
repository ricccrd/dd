# Linux syscall coverage

_Auto-generated by `bash dd-tests/tools/coverage.sh report`. Do not hand-edit; rerun instead._

dd services Linux syscalls in `os/linux/syscall/dispatch.c`, which fans out to one handler
module per family: `svc_{sysv,mem,signal,time,io,fs,proc,net,event,misc,rare}()`. Every
handler keys on the **canonical (aarch64 / asm-generic)** syscall number; the x86-64 frontend
remaps its guest rax through `translate/x86_64/sysmap.h` (`G_NR`) before dispatch, so both
guest arches share one handler set.

## Dynamic is the signal that matters

The raw gap count is not actionable on its own -- most of the 85 are unimplemented on
purpose. What matters is which gaps **real software actually reaches**: each engine logs
`[jit] unhandled syscall N` when a guest hits the ENOSYS default. `coverage.sh dynamic` runs
the compiled guests + busybox applets through both engines and cross-references those hits
against this list, flagging any reached gap as `<-- ACTIONABLE`. Promote a syscall out of
this list when dynamic shows real software needs it, not merely because the kernel defines it.
