# BUG #201 — x86→ARM64 GC-runtime heap-corruption miscompile

Status: **OPEN** (x86_64 guests only; does not affect arm64 guests or the v0.9.6 showcase).
This is the consolidated record of a multi-agent investigation. The live working set (tooling patch,
repro binaries, evidence log) lives under `target/heap201/` (untracked — `target/` is gitignored); this
doc is the tracked summary so the knowledge survives.

## Symptom
A GC'd Go runtime silently corrupts live heap data under dd (x86_64 guest). Canonical repro: a
single-goroutine `map[string]int` + GC/alloc churn (`target/heap201/mcount`, GOMAXPROCS=1). ~40% of runs,
a contiguous insertion-order range of live strings reads back as zeros. On the amd64 oracle and under
`GOGC=off` it ALWAYS passes. Cascades in the wild to **go build, gcc, .NET, clang -O2, CPython-x86,
rustc** — any GC'd/managed runtime. Plain C/C++ (incl. vtables) is unaffected.

## Converged conclusion (mechanism — high confidence)
Go uses the **tiny allocator** for the 10–13B noscan key strings. When it fetches a fresh 16-byte block,
`mallocgc` zeroes it with one 16-byte SIMD store — x86 `movups %xmm15,(%r11)` at mallocgc link ~`0x40cff3`,
lowered by dd to `str q15,[x17]` (ARM `0x3d80022f`). On a corrupting run, the block address (`r11` → dd's
EA `x17`) is a **LIVE tiny block still holding a marked, in-use string**, so this memclr **zeros a live
string in place**. The string header (ptr+len) stays intact; only its bytes go to zero → subsequent map
lookups / `keys[i]` compares fail. Deterministic under `DDFIXHEAP` starting at `firsti=1014` (~410 objects).

Unperturbed ground truth (repro `msweep7`): `curPtr==origPtr`, `curLen==13`, bytes = `\x00`×13; the victim
span is **MARKED and NEVER freed** (sweep inspection every span: setbits=410, allocCount=512). So it is
**not** free-then-reuse — it is a direct wrong-destination memclr while the object is still live.

**Where the wrong destination comes from (still to pin):** the store EA is just `[x17]`, `x17 = r11` = the
block address from the inlined **nextFreeFast** (or the tiny bump `c.tiny+tinyoffset`). So the allocator
handed out a block that is actually a live/marked slot. The value/layout-dependent codegen bug is in the
guest instructions computing that block address — the inlined nextFreeFast / allocCache / freeindex path,
or the tiny bump. It's ASLR/seed-dependent (~40%), which is why every static audit passed (each
computation verifies correct on non-corrupting runs).

## Verified CORRECT — do NOT re-audit (all 0-mismatch, several confirmed on corrupt runs)
- `scanblock` root/stack pointer load (0x41d520; >1.5M loads).
- `typePointers.next` slot returns; `findObject` arenaIndex `(p+0x800000000000)>>26`, pageIndex
  `(p>>13)&0x1fff`, objIndex divMagic `(off*divMul)>>32`.
- `greyobject` mark-store (`lock or %r12b,(%r10)`, bytep = gcmarkBits+objIndex/8) — bit is set.
- `sweep` allocBits/gcmarkBits swap, newMarkBits sizing, countAlloc.
- `refillAllocCache` structure (`allocCache = ^allocBits[word]`); both nextFreeFast inlinings return
  UNMARKED slots on non-corrupt runs.
- Lowerings: `not r64`→`orn`, `bsf/bsr`, `btc`, base `emit_ea_core`, the block-zeroing `movups`→`str q15`
  EA itself (`x17==r11` exactly, no truncation for high heap addrs).
- RULED OUT: async-preempt (`asyncpreemptoff=1`), non-PIE bias fold (`NOGUESTFOLD`), EA-opt (`NOEAOPT`),
  rep-stos/movs (`NOREP`), AES-NI, lazy zero-page guard (`NOLAZYFIX=1 LAZYBUDGET=0`), TSO, FS-base.
- `GOGC=off` fixes it 100% → it is a GC-window bug; a standalone noscan `[]byte` victim is IMMUNE (trigger
  needs scanning/allocating pointer-adjacent objects through the tiny allocator).

## The obstacle (why it's still not pinned)
**Every observation method perturbs the bug away:**
- DDUMP `brk` breakpoints in the GC/alloc hot path slow GC enough that corruption → ~0%.
- DDWATCH RO page-protection on the victim either (a) is far too slow with emulate-and-continue (millions
  of legit tiny-alloc faults → timeout) or (b) shifts the victim off the protected pages (protecting the
  victim moves the manifestation to a different, unprotected object). 0 catches either way.

So the corrupt case is only ever seen WITHOUT instrumentation — which gives the effect but not the
faulting instruction. A **non-breakpoint, non-page-protection** method is required.

## Recommended next approach
**Differential per-alloc trace (lowest-overhead, avoids perturbation):** emit an inline ring-buffer log of
the tiny block addresses `nextFreeFast` returns (from a cheap emitted stub, NOT a signal/breakpoint),
run the SAME deterministic binary (`DDFIXHEAP`) corrupt-vs-clean-vs-amd64-oracle, and find the **first
divergent block address** — that is where the allocator computes wrong.

Suspect instructions to scrutinize in dd's lowering for a value/high-bit-dependent miscompile (the string
span's allocCache word + freeindex/index math):
- the `shr %cl` allocCache update with the `cmp $0x40; sbb; and` shift-by-64 guard (mallocgc
  `0x40cf77-85` / `0x40d139-146`),
- the `bsf` on allocCache,
- the `imul result,elemsize; add base` slot-address (`0x40d163-167`).

## Tooling (in `target/heap201_tooling.patch`, x86 target only, env-gated, INERT by default)
- **`DDFIXHEAP=1`** — deterministic guest heap+image layout: non-PIE bias `0x200000000` (runtime addr =
  link addr + 0x200000000), fixed mmap base 5 TB, image base 8 GB, fixed AT_RANDOM. Makes the corruption
  deterministic (firsti=1014).
- **`DDUMP=<hexRuntimePC>`** — `brk #0x5202` at that guest PC; the SIGTRAP handler in
  `translate/x86_64/elf.c` (`jit86_syncguard`) dumps live guest regs from the ARM ucontext.
- **`DDWATCH=1`** + guest `madvise(addr,len,0x5201)` — mprotects RO; a wrong store faults and
  `jit86_lazyguard` dumps ucontext regs + the ARM store insn + `x17`=EA. Includes `watch_slot_marked`
  (arena walk + mark-bit read) and `watch_emulate_store` (emulate-and-continue, reusing `nonpie_fixup`'s
  ARM decode-and-perform machinery).

mspan offsets (Go 1.22): base+0x18, freeindex+0x30(u16), nelems+0x32(u16), allocCache+0x38, allocBits+0x40,
gcmarkBits+0x48, allocCount+0x60(u16), spanclass+0x67, elemsize+0x68. Arenas L1 ptr for the span walk:
mcount link `0x54c7b8` → biased `0x20054c7b8` (env `DDARENAS`).
