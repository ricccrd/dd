Why dd

A userspace kernel, not a virtual machine

Every other way to run Linux containers on a Mac boots a Linux VM and runs a daemon inside it. dd throws the VM away: the container's compute runs as native Apple-Silicon instructions, and only its syscalls are interpreted — by code that is the kernel.

No virtual machine

No hypervisor, no Linux kernel, no VM to boot and keep resident. The guest's instructions run natively; only the syscall boundary is trapped and serviced in userspace.

Drop-in Docker

dd implements the Docker Engine API, so the ordinary docker CLI drives it. Your existing docker run / ps / images / build muscle memory works unchanged.

The JIT is the kernel

Namespaces, cgroups, overlay image layers and networking are ordinary userspace state — the gVisor / PRoot lineage, with none of a VM's virtualization cost.

aarch64 and x86-64 guests

Run native arm64 Linux images, or x86-64 images via a JIT that decodes x86, synthesizes its flags, and lowers SSE/x87 onto NEON. glibc-class binaries run.

Real container isolation

Overlay layers (copy-up / whiteout), a TOCTOU-free path-jail VFS, PID/UTS/USER namespaces, a private loopback netns, port publishing, and cgroup memory + pids limits.

Desktop app, no root

A native macOS app plus a dd command-line tool install a per-user background daemon and a docker context — everything under your home folder, never sudo.

Why a JIT, not a VM

A container should be a process, not a virtual machine

Every other way to run Linux containers on a Mac — Docker Desktop, Colima, Rancher, OrbStack — boots a Linux VM under a hypervisor and runs the daemon inside it. That VM is a tax you pay all day. dd deletes it: a container is a plain macOS process whose syscalls happen to be serviced by a userspace Linux kernel.

0 GB
Idle memory tax

No VM reserving gigabytes of RAM whether or not a container is running. Memory is per-container and returned to macOS the instant it exits.

No virtiofs
Direct filesystem

Bind mounts are real host paths through a path-jail VFS — no virtiofs/gRPC-FUSE bridge between macOS and a VM, the classic Docker Desktop I/O bottleneck.

Native
CPU, not emulated

The guest's compute runs as native Apple-Silicon instructions. Only the thin syscall boundary is interpreted — there is no hardware-virtualization layer in the hot path.

Resident RAM when idle None — per-container, freed on exit Gigabytes reserved for the VM, always on

Bind-mount / file I/O Direct host filesystem through a path jail virtiofs/gRPC-FUSE bridge across the VM boundary
Port publishing Straight to host sockets Through the VM's NAT/forwarding layer

Footprint to ship & patch No Linux kernel — nothing to CVE-track or update Ships, patches and tracks a whole Linux kernel
Observability A normal macOS process — sample, debug, see it in Activity Monitor An opaque VM; the workload is invisible to host tools

Honest trade-off: a userspace kernel is only as complete as the syscalls it implements, and today dd runs as a single process — great for trusted images. A VM still gives a harder isolation boundary for untrusted code; dd's sentry split is the roadmap answer there.

How it works

Be the kernel, in userspace

dd runs a Linux container by being its kernel. A JIT translates the guest's machine code and traps every syscall instruction; the trap handler is the Linux syscall ABI, implemented against macOS. Everything a real kernel owns becomes state in this one process.

1

Load the ELF

Map the guest's Linux ELF (static-PIE, or dynamic via its ld.so) and build the initial stack with the container's uid/gid.

2

Translate & dispatch

Look up the guest PC; on a miss, JIT-translate the block. Same-ISA code is mostly transliterated; x86-64 is decoded and re-emitted on arm64.

3

Run natively

The translated block executes as native host code until a terminator — a chained branch, an inline-cached indirect jump, or a syscall.

4

Service the syscall

Exit to service() — the Linux ABI on macOS. Every path passes through the container VFS jail; namespaces & cgroups are just state.

Get started

Install once, then it's just Docker

Download the app, drop it in Applications, and run a single setup command. dd registers a background daemon and a docker context for you — no VM, no sudo. The dd command-line tool ships inside the app.

1

Download & install

Open the .dmg and drag dd into Applications. Everything it needs is bundled — no extra tools to install.

2

Set it up

Launch dd once (or run dd install). It starts a per-user background daemon and adds a docker context — all under your home folder, never sudo.

3

Run containers

Point docker at the dd context, or use the app's UI. Your existing images and commands just work.

4

Check it's healthy

dd doctor verifies the socket, daemon and context and prints a fix for anything that needs attention.

Run containers — just Docker

dd registers a dd context, so every ordinary flag flows straight through.

bash
# publish a port, cap memory, run a shell
$ docker --context dd run -p 8080:80 -m 256m alpine \
      sh -c 'echo hi from $(hostname)'

$ docker --context dd ps
$ docker --context dd images
$ docker --context dd run --rm -it ubuntu bash

Manage dd

The dd tool is the control surface for the app and its background daemon.

bash
$ dd install        # daemon + docker context (no sudo)
$ dd app            # open the desktop app
$ dd doctor         # health check + fixes
$ dd daemon status  # start | stop | restart | logs
